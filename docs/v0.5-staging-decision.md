# v0.5 Safe Staging Contract

Status: accepted for the first v0.5 implementation slice. This document is
normative for follow-up implementation issues but does not itself authorize
copying, retagging, renaming, or any source-library mutation.

Music Manager staging remains local-first and review-driven. It reads one
finalized schema 1 scan plus an explicit user approval file, copies approved
source files into a separate staging set, verifies those copies with content
checksums, and records durable local evidence. The source library is never
modified.

## Goals and terminology

Version 0.5 introduces a safe staging library for future organization work.
It does not organize, rename, retag, normalize albums, fetch artwork, or apply
MusicBrainz metadata.

The local subjects are:

- a `scan run`: one finalized `reports/<scan-id>/` artifact set;
- a `source root`: the user-selected root that the scan paths are relative to,
  the same scan root supplied to `scan --source`;
- a `staging root`: a separate local directory selected by the user;
- a `stage set`: one exclusive directory below the staging root, identified by
  `stage_id`;
- an `approval file`: a user-controlled CSV that names inventory rows approved
  for staging; and
- a `staged copy`: a byte-for-byte copy under the stage set.

The word `staged` means only that a verified local copy exists in the stage
set. It does not approve later rename, tag, artwork, deletion, upload, or
promotion work.

## Command and approval boundary

Staging is a separate operation against an explicit finalized scan:

```text
music-manager stage plan \
  --scan-run reports/<scan-id> \
  --approval approved-files.csv

music-manager stage apply \
  --scan-run reports/<scan-id> \
  --stage-id <stage-id> \
  --source-root /path/to/music \
  --staging-root /path/to/staging \
  --approve
```

`stage plan` writes reviewable local plan evidence only. It must not read,
hash, copy, create, delete, overwrite, or modify any source-library or staging
file. It validates the scan run, approval file, and planned relative staging
paths before registering the plan.

`stage apply` is the only v0.5 operation that copies bytes. It requires all of:

- the `stage apply` command;
- a finalized schema 1 scan run;
- an existing registered staging plan for the requested `stage_id`;
- an explicit `--source-root`;
- an explicit `--staging-root` that passes destination policy; and
- the exact `--approve` flag.

There is no implicit approval from MusicBrainz match status, duplicate reports,
configuration defaults, previous stage runs, cache contents, terminal prompts,
or the presence of a staging plan. Without `--approve`, `stage apply` fails
before opening source paths or creating staging files.

Before any copy begins, the terminal preflight states that Music Manager will
read approved source files under the selected source root, write verified
copies under the selected staging root, and leave the source library unchanged.
It prints counts and the stage set path. It must not print private checksums;
checksums are recorded only in the generated staging reports.

## Approval file

The approval file is UTF-8 CSV with this exact ordered header:

```text
scan_id,file_record_id,decision
```

`scan_id` must match the selected scan. `file_record_id` must identify exactly
one row in `library_scan.csv`. `decision` is either `stage` or `skip`.
Only `stage` rows enter the plan. Duplicate file IDs, unknown file IDs, mixed
scan IDs, unknown decisions, malformed CSV, and empty staged selections are
errors. The approval file is input only and is never modified.

Approval rows do not authorize source-library mutation. They authorize only
copying the selected readable audio rows into the requested stage set.

## Eligibility

Only scan rows with `file_type: audio` and `record_status: ok` are eligible.
Archives, unreadable audio rows, missing inventory rows, and rows from legacy
v0.2 reports are not eligible. MusicBrainz status is not eligibility. Duplicate
candidate status is not eligibility.

An approval row with decision `stage` that identifies an ineligible scan row is
not a plan-level error; `stage plan` records it with `plan_status`
`not_eligible` and a reason code, and it never enters copy execution.

The source path for a row is always `source_root / row.path`, where `row.path`
is the validated source-relative path from the scan artifact. Paths remain
opaque report data until the user supplies `--source-root` for apply.

`stage apply` rejects a source file when:

- the resolved source path escapes the source root;
- any source path component is a symlink;
- the source path is missing or is not a regular file;
- the current size or modified time no longer matches the scan fingerprint
  when that fingerprint is available;
- the file cannot be read; or
- the source content changes between the checksum read and the copy read, which
  surfaces as a digest mismatch during staged copy verification.

Rows rejected during apply become staging errors. They do not authorize retries
against a different source path or relaxed path policy in the same stage set.
These per-row source checks are distinct from the command-level apply preflight.
A command-level preflight failure registers no artifacts and aborts the run; a
per-row failure records one staging error and does not stop remaining rows.

## Staging root and layout

The staging root is an explicit local path supplied by the user. It is never
stored in scan artifacts, staging reports, or manifests. The implementation may
compare resolved paths during preflight but must persist only source-relative
and stage-set-relative paths.

The source root and staging root must be distinct. The staging root must not be
inside the source root, and the source root must not be inside the staging
root. This prevents recursive scans, accidental promotion, and path confusion.

Each plan reserves one `stage_id`. `stage apply` creates one exclusive stage
set for that registered plan:

```text
<staging-root>/<stage-id>/
  staging_manifest.json
  files/
```

`stage_id` is a lowercase canonical UUIDv4 generated before the plan is
written. Apply preflight verifies that `<staging-root>/<stage-id>` does not
already exist, then creates it exclusively before any copy begins. Files are
staged under:

```text
files/<source-relative path>
```

The source-relative spelling is preserved exactly except for the fixed `files/`
prefix. No path normalization, artist-folder guessing, MusicBrainz naming,
case folding, Unicode normalization, extension rewriting, or collision suffix
is applied in v0.5. Because schema 1 inventory paths are unique and
source-relative, this layout is deterministic and collision-free within one
stage set.

Staging never overwrites an existing file. If any destination path or parent
inside the stage set exists unexpectedly or is a symlink, that row fails with a
staging error. Temporary files use an implementation-owned suffix in the stage
set and are ignored by readers.

## Checksums and copy verification

Version 0.5 introduces content checksums only for staging, recorded only in the
new staging artifacts. It does not add content checksums to the existing scan,
analysis, or MusicBrainz artifact families and does not reinterpret the schema 1
stat fingerprint as content identity.

For each approved row, `stage apply` streams the source file through SHA-256,
copies bytes to a temporary file in the stage set in a separate source read,
atomically moves the staged temporary file into place, then streams the staged
copy through SHA-256. The checksum read and the copy read must be distinct so
that a source file changing during staging surfaces as a digest mismatch. The
copy is verified only when:

- source and staged byte counts match;
- source and staged SHA-256 digests match; and
- the staged path remains a regular file under the stage set.

A verification mismatch is a staging error. The implementation should remove
the failed staged file when it can do so without deleting any preexisting user
file. It must never delete or modify the source file.

## Artifact set

Version 0.5 adds three scan-run artifacts:

```text
reports/<scan-id>/
  staging_plan.csv
  staging_copies.csv
  staging_errors.csv
```

The plan artifact may be registered alone after `stage plan`. The copy and
error artifacts are registered together after `stage apply`, even when some
approved rows fail. A failed preflight registers no new artifacts. A failed
apply that starts copying registers the completed evidence atomically before
exiting nonzero.

Rerunning `stage plan` creates a new `stage_id` and replaces only the registered
plan artifact when the previous plan has not been applied. Once any copy or
error artifact exists for a `stage_id`, that stage set is immutable. A new
attempt requires a new plan and a new stage set. The scan manifest records only
the latest staging artifact family, so planning again after an applied stage set
deregisters the earlier family from the manifest; older stage set directories
remain local evidence under their staging roots and are not reused.

The exact ordered headers are:

```text
staging_plan.csv
scan_id,stage_id,file_record_id,source_path,stage_relative_path,plan_status,reason_code

staging_copies.csv
scan_id,stage_id,file_record_id,source_path,stage_relative_path,source_size_bytes,source_sha256,staged_size_bytes,staged_sha256,copy_status

staging_errors.csv
scan_id,stage_id,file_record_id,source_path,stage,error_code,message
```

`source_path` is the schema 1 source-relative path. `stage_relative_path` is
relative to the stage set and starts with `files/`. It follows the same path
privacy and syntax rules as schema 1 relative paths. `plan_status` is `planned`
or `not_eligible`. `copy_status` is `verified`. Error `stage` is one of
`source_preflight`, `copy`, `verification`, or `finalization`. Error codes are
stable lowercase snake case.

Plan and copy rows sort by source path. Error rows sort by source path, then
stage, then error code. Empty reports still contain their header. Every CSV row
contains `scan_id` and `stage_id`.

## Staging manifest

Each stage set owns a private `staging_manifest.json` under the stage set
directory. It is local operational evidence, not a scan artifact, and is not
registered in `scan_manifest.json`.

The staging manifest records:

- staging schema version;
- application version;
- `scan_id`;
- `stage_id`;
- state: `copying`, `complete`, `incomplete`, or `failed`;
- UTC `started_at` and `completed_at`;
- counts for planned, verified, and failed rows;
- SHA-256 digests for the staging CSV reports copied into the stage set, if
  the implementation mirrors them there; and
- sanitized configuration values.

It must not record source root, staging root, approval-file path, config path,
hostnames, usernames, environment variables, MusicBrainz query text, or raw
terminal arguments.

## Schema compatibility

These artifacts are a backward-compatible schema addition, so v0.5 introduces
schema `1.2.0`. Version 0.5 readers accept schema `1.0.x`, `1.1.x`, and
`1.2.x`. Registering staging artifacts into a validated earlier scan run
atomically upgrades its manifest to `1.2.0` without rewriting primary scan,
analysis, or MusicBrainz CSVs.

Derived artifact families remain independent. Rerunning analysis must not
replace staging artifacts. Rerunning MusicBrainz matching must not replace
staging artifacts. Staging must not replace analysis or matching artifacts.

## Privacy and safety

Staging reports may contain private relative paths, filenames, metadata-derived
folder names, and content checksums. They remain local and Git-ignored under
the existing report policy.

The following values must never be stored in scan artifacts or staging reports:

- absolute source root;
- absolute staging root;
- approval-file path;
- config path;
- usernames, hostnames, home directories, or environment variables;
- source-library symlink targets;
- source-library absolute paths; and
- credential, account, token, cookie, or key material.

Staging performs no network access. It must not instantiate or call the
MusicBrainz client. It must not read audio tags for decision making; it relies
on the validated scan inventory and then reads approved file bytes only for
copying and verification.

## Failure behavior

A plan failure leaves no registered staging plan and no stage set directory.

An apply preflight failure leaves source and staging files untouched and
registers no copy or error artifacts. An apply failure after copying begins
keeps verified staged files, records row-level errors, finalizes the stage set
as `incomplete`, registers copy and error artifacts atomically, and exits
nonzero. A fatal failure that prevents normal finalization marks the staging
manifest `failed` when the manifest can still be written; a run that dies before
finalizing leaves the transient `copying` state.

Fatal finalization failures should leave enough local evidence for manual
inspection, but consumers read only registered scan artifacts and finalized
staging manifests. Readers must reject `copying` and `failed` stage sets for
future organization work. Within an accepted stage set, `staging_copies.csv` is
the authoritative file list: readers must consume only files recorded there with
`copy_status` `verified` and must ignore any other stage-set content.

## Offline testing requirements

Follow-up implementation must include:

- approval CSV validation for header, decisions, duplicates, mixed scan IDs,
  missing rows, legacy input, and empty selections;
- plan-only tests proving no source or staging file is opened, hashed, copied,
  overwritten, or deleted;
- apply tests requiring `--approve` before any source or staging access;
- source-root and staging-root ancestor rejection;
- missing source, changed stat fingerprint, unreadable source, symlink source,
  symlink destination, and existing destination failures;
- successful SHA-256 verification with deterministic staged layout;
- injected copy and checksum mismatch failures that never mutate the source;
- atomic artifact registration and schema 1.0/1.1-to-1.2 manifest upgrade;
- preservation of existing analysis and MusicBrainz artifacts;
- privacy tests seeded with absolute roots, usernames, hostnames, config paths,
  symlink targets, and approval-file paths proving they are not persisted;
- no MusicBrainz, DNS, socket, or other network calls;
- no access to unapproved source files; and
- at least 100,000 generated approval rows to detect quadratic behavior without
  committed media or wall-clock assertions.

Synthetic fixtures use tiny generated files and invented metadata only.
Committed real music, copyrighted artwork, ZIP archives, private reports, and
recorded production responses are excluded.

## Follow-up issue boundaries

With this document accepted, v0.5 implementation work should remain split:

1. add schema 1.2 staging artifact models and strict validators;
2. add approval-file parsing and deterministic plan generation;
3. add staging-root preflight and stage-set manifest lifecycle;
4. add checksum-verified copy execution with row-level errors;
5. wire the `stage plan` and `stage apply` CLI with safety disclosures; and
6. prepare v0.5.0 documentation, changelog, version, tag, and release notes.

Changes to artifact fields, approval semantics, checksum policy, path policy,
or source/staging mutation boundaries require an explicit amendment rather than
an implementation-only decision.

## Exclusions

Version 0.5 adds no source-library mutation, deletion, move, rename, retagging,
artwork lookup, album normalization, MusicBrainz application, duplicate
resolution, release-edition selection, acoustic fingerprinting, upload,
background processing, inbox automation, or organization of staged copies.
