# v0.4 MusicBrainz Matching Contract

Status: released in v0.4.0 after acceptance in
[issue #34](https://github.com/saari-co/music-manager/issues/34). This document
remains the normative contract for the released matching behavior.

MusicBrainz matching remains read-only and report-driven. It consumes one
finalized schema 1 scan, never opens a source-library path, and never applies
metadata. Network access is disabled by default and requires explicit user
consent.

## Goals and terminology

Version 0.4 produces reviewable candidates for two local subject types:

- an `album` is a deterministic group of scan rows and is compared with
  MusicBrainz release groups; and
- a `recording` is one scan row and is compared with MusicBrainz recordings.

A release group represents the album-level work across editions. Version 0.4
does not claim to identify a specific release edition because the schema 1
inventory has no disc ID, barcode, or catalog number. A later contract may add
release matching when it has evidence sufficient to distinguish editions.

The word `matched` in an output status means only that one candidate cleared the
documented score and margin thresholds. It is not a user confirmation and does
not authorize tag, filename, artwork, or library changes.

## Command and consent boundary

Matching is a separate operation against an explicit run:

```text
music-manager match \
  --scan-run reports/<scan-id> \
  --musicbrainz
```

The `match` command and affirmative effective consent are both required.
Consent may come from the CLI flag or local config described below. A scan,
analysis, application startup, cache operation, or config load must never
contact MusicBrainz. There is no implicit consent based on previous use, an
existing cache, or the presence of MusicBrainz identifiers.

The effective configuration has `musicbrainz.enabled: false` by default. A
local config may set it to `true` as persistent explicit consent; the CLI
`--musicbrainz` flag enables it for one invocation and `--no-musicbrainz`
overrides persistent consent. The command must report whether consent came from
config or CLI before the first request.

Before any network operation, the command validates consent, the selected scan
artifact set, client configuration, and output destination. Invalid or legacy
input fails without opening the cache or transport. Only `complete` and
`incomplete` schema 1 runs are accepted. Unversioned v0.2 reports are excluded
because they have no durable scan or file identity.

The terminal preflight states that normalized artist, album, and title text may
be sent to MusicBrainz. It also states that paths, filenames, scan IDs, file
record IDs, and audio content are not sent.

## Application identification

Every live request uses this shape:

```text
music-manager/<application-version> (https://github.com/saari-co/music-manager)
```

The version is the installed Music Manager version. The project URL is the
maintainer contact and is not user-configurable in v0.4. User email addresses,
hostnames, usernames, config paths, and library data never enter the
`User-Agent`.

The client uses only HTTPS `GET` requests to
`https://musicbrainz.org/ws/2/`, requests JSON explicitly, follows no
cross-origin redirect, and uses no MusicBrainz account or authentication.
Production endpoint overrides are excluded. Tests inject a transport instead
of changing the production origin.

This follows MusicBrainz's published requirement for a meaningful application
name, version, and maintainer contact:
<https://musicbrainz.org/doc/MusicBrainz_API/Rate_Limiting>.

## Client boundary

Matching depends on an application-owned client interface with two operations:

```text
search_release_groups(album_artist, album_title, limit)
search_recordings(track_artist, track_title, limit)
```

The interface returns validated, application-owned value objects. HTTP library
types and raw MusicBrainz response dictionaries do not cross the boundary.
Production and fake clients implement the same interface.

The production client owns request construction, JSON decoding, response
validation, cache lookup and storage, rate limiting, retry behavior, and
sanitized errors. Its transport, monotonic clock, wall clock, and sleeper are
injectable. Candidate generation owns grouping, scoring, classification, and
artifact rows. No layer opens paths found in scan artifacts.

Each search uses a limit of 10. Queries use fielded, quoted Lucene terms:

```text
releasegroup:"<album>" AND artist:"<album artist>"
recording:"<title>" AND artist:"<track artist>"
```

Lucene metacharacters are escaped before URL encoding. Optional local fields
do not narrow the search in v0.4; they are scoring evidence instead. This keeps
candidate retrieval separate from confidence evaluation. Response items that
lack a valid canonical MusicBrainz UUID or required display text are ignored
and counted as malformed response items. Duplicate MBIDs collapse to the item
with the highest search score, then the lexically smallest normalized
candidate-field tuple.

The relevant MusicBrainz search fields and JSON response format are documented
at <https://musicbrainz.org/doc/MusicBrainz_API/Search>.

## Cache policy

The production client always uses a persistent, versioned cache in the
operating system's per-user cache directory. The default logical location is:

```text
music-manager/musicbrainz-v1.sqlite3
```

An explicit cache-path override is allowed for testing and advanced local use,
but its absolute path is never persisted in reports or the scan manifest. The
cache directory and database use user-only permissions where the platform
supports them.

The cache contract is:

- a key is SHA-256 over the UTF-8 canonical origin, API path, response format,
  and sorted encoded query parameters;
- raw request URLs and query text are not stored as cache keys or diagnostic
  columns;
- only an HTTP 200 response that decodes as valid JSON is stored;
- errors, redirects, partial bodies, and malformed JSON are never cached;
- an entry is fresh for exactly 30 days, or 2,592,000 seconds, from successful
  response completion;
- expired or corrupt entries are misses and are replaced only by a successful
  live response;
- stale data is not used after a failed refresh; and
- a cache hit performs no sleep and consumes no rate-limit slot.

Cached response bodies contain MusicBrainz's public response data. They are
local implementation state, not scan artifacts, and are not registered in
`scan_manifest.json`. Cache writes are transactional and safe to abandon
without affecting a completed matching artifact set.

## Rate limiting, timeouts, and retries

All live attempts are serialized. A shared monotonic gate enforces at least
1.1 seconds between request start times, including retries. The first live
attempt may start immediately. There is no random jitter or concurrency in
v0.4, so a fake clock can reproduce the schedule exactly.

Each attempt has a 30-second total timeout. The client makes at most four
attempts for:

- connection failures and timeouts;
- HTTP 429, 500, 502, 503, and 504; and
- a truncated or malformed JSON response.

The deterministic delays before attempts two through four are 1, 2, and 4
seconds. The effective delay is the greatest of the backoff delay, the
rate-limit gate, and a valid `Retry-After` value. Both delta-seconds and
HTTP-date forms use injected clocks in tests. A `Retry-After` longer than 60
seconds ends the operation with an error instead of retrying earlier than the
service requested. Other 4xx responses, an unsafe redirect, and response
contract violations are not retried.

Retry exhaustion produces an `error` result, never an `unmatched` result.
After one operation exhausts retries, a run-level circuit prevents further live
attempts; fresh cache hits may still be used and uncached subjects become
`error` results. Any run containing an error exits nonzero after atomically
registering its report set. This preserves successful evidence, avoids
repeatedly contacting an unhealthy service, and makes incomplete assessment
visible.

MusicBrainz currently documents an average source-IP limit of one request per
second and uses HTTP 503 when throttling:
<https://musicbrainz.org/doc/MusicBrainz_API/Rate_Limiting>. The 1.1-second
client interval is deliberately conservative. A future service-policy change
requires a reviewed contract amendment.

## Local subjects and identity

Only `file_type: audio` rows participate. Archives have no matching subject.
Every audio row receives a recording result row, including `not_eligible` for
an unreadable row or missing required metadata.

A recording is eligible when `record_status` is `ok` and normalized `artist`
and `title` are both non-empty. Its subject identity is the existing
`file_record_id`. Album, duration, release year, and track/disc fields may add
evidence but are not eligibility requirements.

Metadata normalization is Unicode NFKC, case folding, surrounding-trim, and
internal whitespace collapse. Punctuation is preserved. Query text is NFKC
normalized and whitespace-collapsed but preserves display case before Lucene
escaping. When an album group contains multiple display spellings with the
same normalized value, its query uses the lexical minimum by
`(casefolded query text, query text)`.

An album's effective artist is `album_artist` when present, otherwise `artist`.
Eligible readable audio rows with a non-empty effective artist and album are
grouped by:

```text
(normalized effective artist, normalized album)
```

This intentionally does not guess that different track artists with a blank
album artist form one compilation. Those rows remain eligible for recording
matching, and each distinct artist/album tuple remains a separate album group.

`album_group_id` is canonical UUIDv5 using the scan UUID as namespace and this
UTF-8 name:

```text
musicbrainz-album-group-v1\0<normalized effective artist>\0<normalized album>
```

All non-empty release years in a group are year evidence only when they agree.
Conflicting years make album year evidence unavailable rather than selecting
one. Matching reads only the validated inventory; paths are opaque and are
never grouping or query evidence.

## Artifact set

A completed match atomically registers these four derived artifacts in the
selected scan directory:

```text
musicbrainz_album_groups.csv
musicbrainz_album_candidates.csv
musicbrainz_recording_candidates.csv
musicbrainz_match_results.csv
```

The artifact entry configuration records only allowlisted values: client policy
version, scoring model version, candidate limit, thresholds, cache maximum age,
rate interval, retry count, timeout, and whether network consent came from CLI
or config. It records no query text, cache path, environment variable, or raw
configuration.

The exact ordered headers are:

```text
musicbrainz_album_groups.csv
scan_id,album_group_id,file_record_id

musicbrainz_album_candidates.csv
scan_id,album_group_id,candidate_rank,release_group_mbid,title,artist_credit,
first_release_date,primary_type,secondary_types_json,musicbrainz_search_score,
title_similarity,artist_similarity,year_similarity,confidence_score

musicbrainz_recording_candidates.csv
scan_id,file_record_id,candidate_rank,recording_mbid,title,artist_credit,
duration_ms,first_release_date,matched_release_mbid,matched_release_title,
musicbrainz_search_score,title_similarity,artist_similarity,
duration_similarity,album_similarity,confidence_score

musicbrainz_match_results.csv
scan_id,subject_type,subject_id,status,candidate_count,top_candidate_mbid,
top_confidence_score,confidence_margin,reason_code
```

Album membership rows expose the join from a deterministic album group to scan
records without repeating local metadata. Candidate fields contain the
whitelisted MusicBrainz values and every factor used by the score.
`secondary_types_json` is a compact JSON array of unique strings in lexical
order. A nullable value is an empty CSV cell; an empty collection is `[]`.
`artist_credit` concatenates credited names and join phrases in MusicBrainz
order. Candidate text is NFKC-normalized and trimmed without changing case.
Dates use MusicBrainz's validated `YYYY`, `YYYY-MM`, or `YYYY-MM-DD` precision.

Candidate rank starts at 1. Candidates sort by descending confidence, then
descending MusicBrainz search score, then lexical MBID. Artifact rows sort by
subject type and subject UUID, with candidate rank inside a subject. The same
validated input and fake responses must produce byte-identical CSV files
regardless of input row or response item order.

For recording album evidence, equal release-title similarities select the
lexically smallest release MBID. `candidate_count` is the number of valid,
deduplicated candidates emitted for the subject and is never greater than 10.

CSV representation and validation otherwise follow the schema 1 contract.
UUID fields are lowercase canonical UUID text. Similarities are decimals from
0 through 1 with four fractional digits. Confidence and margin are decimals
from 0 through 100 with two fractional digits. MusicBrainz search score is an
integer from 0 through 100. `duration_ms` is a non-negative integer.

## Schema compatibility and artifact lifecycle

These additional artifact names are a backward-compatible schema addition, so
v0.4 introduces schema `1.1.0`. Version 0.4 readers accept both `1.0.x` and
`1.1.x`. Registering matching artifacts into a validated v0.3 run atomically
upgrades its manifest to `1.1.0` without rewriting the primary CSVs.

Derived artifact families are independent. Rerunning ordinary analysis
replaces only analysis artifacts; rerunning matching replaces only the four
MusicBrainz artifacts. A failure before registration leaves the previous
complete family registered and removes new temporary files. A completed run,
including one with subject-level `error` results, replaces the prior matching
family as one atomic set.

All four artifacts share one generation timestamp and matching configuration
in their manifest entries. Their hashes and row counts use the existing
manifest rules. Matching does not change the scan's `complete` or `incomplete`
state because that state describes primary scan completion.

## Evidence and confidence model

The scoring model identifier is `musicbrainz-confidence-v1`. MusicBrainz's
search score is upstream retrieval evidence, not Music Manager confidence.
It contributes only five points.

Text similarity uses normalized Levenshtein similarity:

```text
1 - edit_distance(local, candidate) / max(length(local), length(candidate))
```

Two equal normalized strings score 1. If either value is unavailable, the
factor scores 0. Length counts Unicode code points after normalization. Raw
factors use exact decimal arithmetic and round half-up to four decimal places;
the weighted score uses those serialized factor values and rounds half-up to
two decimal places.

Album candidate confidence is:

```text
45 * title_similarity
+ 35 * artist_similarity
+ 15 * year_similarity
+  5 * (musicbrainz_search_score / 100)
```

Year similarity is 1 for the same year, 0.75 for a one-year difference, 0.50
for two years, 0.25 for three years, and 0 otherwise. Missing or conflicting
local year evidence scores 0.

Recording candidate confidence is:

```text
40 * title_similarity
+ 30 * artist_similarity
+ 20 * duration_similarity
+  5 * album_similarity
+  5 * (musicbrainz_search_score / 100)
```

Duration similarity is `max(0, 1 - absolute_difference_seconds / 10)`.
Unavailable local or candidate duration scores 0. Album similarity is the
greatest text similarity between the local album and release titles included
with the recording search result. The selected release MBID and title are
recorded beside that evidence; missing album or release context scores 0.

The confidence bands are:

- `high`: 85.00 through 100.00;
- `medium`: 60.00 through 84.99; and
- `low`: below 60.00.

These bands are explanatory and are not emitted as a separate field because
they are derived exactly from `confidence_score`.

## Result classification

`musicbrainz_match_results.csv` has exactly one row for each eligible album
group and each audio recording subject. Its status is one of:

- `matched`: the top candidate is at least 85.00 and leads the second candidate
  by at least 10.00 points;
- `ambiguous`: the top candidate is at least 60.00 but does not meet both
  `matched` conditions;
- `unmatched`: there is no candidate or the top candidate is below 60.00;
- `not_eligible`: a recording's required local metadata or readable scan state
  is absent; or
- `error`: the subject was not assessed because its client operation failed.

For a single candidate, the comparison score is zero. `confidence_margin` is
top minus comparison score. Top-candidate fields and margin are empty when
there is no candidate. Every candidate is reported even when its confidence is
low.

Stable `reason_code` values explain the classification:

```text
high_confidence_with_margin
high_confidence_close_candidates
medium_confidence
below_confidence_threshold
no_candidates
missing_artist
missing_title
unreadable_record
request_failed
malformed_response
```

When more than one ineligibility reason applies, precedence is unreadable
record, artist, then title. If a non-empty response contains no valid candidate,
the result is `error`/`malformed_response`; malformed items alongside valid
candidates are counted in the terminal summary. A result status is never
silently upgraded by a MusicBrainz search score or by the presence of an MBID
alone.

## Privacy and observability

The only local values allowed in an outgoing query are the normalized artist,
album, and title values described above. The transport must never receive:

- a path or filename;
- scan, file-record, album-group, or fingerprint identifiers;
- file timestamps, sizes, codec, bitrate, or other technical metadata;
- config contents, cache paths, usernames, or hostnames; or
- audio, artwork, checksums, or acoustic fingerprints.

MusicBrainz and the network operator can still observe the query text, source
IP, timing, and application `User-Agent`. Opt-in text must not describe the
operation as anonymous.

Default logs and terminal errors omit query text and response bodies. They may
include the MusicBrainz operation, HTTP status, retry count, and a sanitized
error category. Candidate reports remain private and Git-ignored under the
existing report policy. There is no telemetry, analytics, background polling,
or scheduled refresh.

## Offline testing requirements

All automated tests run with a fake client or injected fake transport. No test
may depend on DNS, a socket, current MusicBrainz data, wall-clock sleeps, or
committed private library metadata. Follow-up implementation must cover:

- opt-out through defaults, config, and CLI overrides with zero transport calls;
- preflight validation failure before cache or transport access;
- exact `User-Agent`, HTTPS origin, JSON request, query escaping, and redirect
  policy;
- fresh, expired, corrupt, and repeated cache entries, including proof that
  cache keys do not contain raw query text;
- deterministic rate-gate, timeout, retry, backoff, and `Retry-After` behavior
  with fake clocks;
- retryable and non-retryable HTTP, transport, and response failures;
- release-group and recording response fixtures with missing optional fields,
  malformed items, duplicates, and reordered results;
- exact album grouping IDs and membership, recording eligibility, scoring,
  rounding, ranking, tie-breaking, confidence margins, and all statuses;
- source row and response reordering with byte-identical CSV output;
- strict headers, values, cross-artifact IDs, hashes, and schema 1.0-to-1.1
  manifest upgrade;
- all-or-nothing replacement of only the matching artifact family;
- no source-library path access during matching; and
- seeded private paths and identifiers proving that outgoing requests, cache
  keys, logs, and errors contain only allowlisted data.

Synthetic fixtures use invented artists, albums, titles, and canonical UUIDs.
Recorded production responses and copyrighted media are not fixtures.

## Implementation history and change boundary

The contract was implemented through reviewed schema/client, offline-policy,
production-client, subject/retrieval, scoring/classification,
artifact-registration, and CLI-orchestration slices. Each slice preserved the
network, privacy, and no-application boundaries in this contract.

Changes to fields, scoring, thresholds, cache semantics, or data sent over the
network require an explicit contract amendment rather than an
implementation-only decision.

## Exclusions

Version 0.4 adds no metadata application, user-confirmed selection state,
retagging, renaming, moving, copying, deletion, staging, artwork lookup,
acoustic fingerprinting, AcoustID, disc-ID submission, authenticated
MusicBrainz operation, collection editing, release-edition selection,
background refresh, polling, or source-library access.
