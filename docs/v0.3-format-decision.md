# v0.3 Durable Scan and Report Contract

Status: proposed for issue #12. This is normative and does not authorize implementation.

The contract remains local-first and read-only for the source library, with no
network access. Report files are the only writes. Any future library mutation
must remain separately planned, previewed, and explicitly approved.

## Artifact set and relationships

One scan owns one immutable ID and one directory:

```text
reports/<scan-id>/
  scan_manifest.json
  library_scan.csv
  scan_errors.csv
  library_analysis.csv
  duplicate_candidates.csv
  missing_metadata.csv
  corrupt_files.csv
  quality_summary.csv
```

The first three files are primary scan artifacts. Optional analysis CSVs are
generated without opening library files. All versioned artifacts stay together.

`scan_manifest.json` is the authority for identity, schema, provenance, state,
counts, and integrity. `library_scan.csv` has one row per discovered supported
audio file or archive, including metadata failures. `scan_errors.csv` records
structured findings, including paths for which no inventory row could be made.

Every CSV row contains the manifest's `scan_id`; file-oriented rows also carry
`file_record_id` when an inventory row exists. An analyzer rejects mixed scan
IDs. Successful analysis atomically registers its complete derived set without
changing scan state; failed analysis registers and leaves no partial set.

## Manifest

`scan_manifest.json` is UTF-8 JSON and has these required top-level fields:

| Field | Contract |
| --- | --- |
| `schema_version` | Semantic version of the entire artifact contract, initially `1.0.0`. |
| `application_version` | Music Manager version that performed the scan. |
| `scan_id` | Lowercase canonical UUIDv4 generated once before discovery. |
| `state` | `running`, `complete`, `incomplete`, or `failed`. |
| `started_at` | UTC RFC 3339 timestamp. |
| `completed_at` | UTC RFC 3339 timestamp for a final state; otherwise `null`. |
| `artifacts` | Entries keyed by logical artifact name. |
| `counts` | Inventory rows, finding rows by severity, and skipped symlink count. |
| `configuration` | Sanitized effective scan configuration. |

Each finalized artifact entry contains its relative filename, role (`primary`
or `derived`), producer version, generation time, row count, and SHA-256 digest.
A derived entry also contains sanitized analysis configuration. The manifest
does not hash itself.

The manifest starts at `running` and is replaced atomically as artifacts
finalize. Consumers read only `complete` or `incomplete` runs. An interrupted
`running` directory is abandoned and unusable, not silently `incomplete`.

## Directory and completion behavior

- `reports/<scan-id>/` is created exclusively, never reused or overwritten.
- Writes use temporary files in that directory and atomic replacement; readers
  ignore temporary files.
- `complete`: traversal and both primary CSVs finished with no `error` or
  `fatal` findings; informational findings are allowed.
- `incomplete`: traversal and a usable inventory finished with one or more
  recoverable `error` findings and no fatal findings.
- `failed`: a fatal condition prevented a usable inventory. The manifest and a
  fatal error row are best effort; `library_scan.csv` may be absent and is not
  consumable.
- No `latest` file or symlink is created. Selection of a prior run is explicit.

## CSV contracts

Versioned CSVs are UTF-8 without BOM, comma-delimited, header-first, and use
standard quoting. Empty cells represent nullable values only. Rows use
deterministic source-relative path order.

The schema 1 `library_scan.csv` columns are:

```text
scan_id, file_record_id, file_fingerprint, path, extension, file_type,
file_size_bytes, modified_time_ns, artist, album_artist, title, album, date,
release_year, track_number, track_total, disc_number, disc_total, genre,
composer, is_compilation, codec, container, bitrate_kbps, duration_seconds,
sample_rate_hz, bit_depth, channels, record_status
```

This preserves core fields, separates raw `date` from parsed `release_year` and
track/disc numbers from totals, and adds common metadata and technical fields.
Tag text is preserved except surrounding whitespace; analysis normalizes only
in memory. `record_status` is `ok` or `error`, with details in the error CSV.

The schema 1 `scan_errors.csv` columns are:

```text
scan_id, file_record_id, path, stage, severity, error_code, message
```

`file_record_id` and `path` may be empty only for failures before they existed.
`stage` and `error_code` are stable machine values. `severity` is `info`,
`error`, or `fatal`; `message` is sanitized text. An empty report has a header.

Each analysis CSV adds `scan_id` as its first column. File-oriented analysis
reports also add `file_record_id`; aggregate reports do not. This makes derived
rows joinable to exactly one scan without treating a path as identity.

## Path privacy and configuration

Versioned artifacts never persist an absolute scan root, home, working
directory, config location, or library path. Durable output is source-relative;
the v0.2 absolute option is legacy behavior, not schema 1.

Every `path` preserves filesystem spelling relative to the selected root, with
`/` separators. It is non-empty, has no absolute prefix, `.` or `..` segment,
and is not Unicode- or case-normalized. Consumers treat it as opaque unless an
explicit workflow binds it to a user-selected root.

Errors follow the same policy. Relative names and tags can still be private, so
reports remain local and Git-ignored; portability is not anonymity.

The configuration snapshot is the allowlisted effective value after defaults,
config, and CLI overrides. Schema 1 records source-relative ignore patterns,
`path_mode: "relative"`, and `follow_symlinks: false`. It excludes raw YAML,
environment variables, unknown keys, secrets, and absolute paths.

## Symlinks

Schema 1 never follows symlinks. Link-aware discovery does not open or
stat-follow supported-file links. Directory and supported-file links produce an
`info`/`symlink_skipped` finding containing only the relative link path. These
expected findings do not make a scan incomplete.

Future opt-in requires a reviewed amendment covering root escape, cycles,
duplicate traversal, and target privacy; it is not a schema 1 option.

## Record identity and change detection

`file_record_id` is UUIDv5 with the scan UUID as namespace and
`"file-record-v1\0" + path` as name. It is deterministic and unique within one
scan and copied into derived rows. It changes in a new scan and is not musical
identity, a duplicate decision, an external ID, or a content checksum.

`file_fingerprint` is `stat-v1:` plus SHA-256 of UTF-8
`size=<decimal>\nmtime_ns=<decimal>\n`, or empty if either value is unavailable.
It cheaply compares the same path across scans but reads no content and proves
no equality: preserved stats can hide change and different files can match. A
future content checksum requires a separate field and contract.

## Versioning, validation, and legacy reports

Schema versions use semantic versioning: major for incompatible field or
meaning changes, minor for backward-compatible additions, and patch for
clarifications that do not change serialized data.

Readers support explicit versions and reject unsupported major or newer minor
versions before CSV parsing. Each version requires its exact ordered header and
valid row width, required values, types, enums, IDs, paths, and cross-artifact
scan IDs. Missing, duplicate, or extra columns and invalid values are errors,
never coerced. Older-version adapters are explicit; header guessing is banned.

An unversioned CSV with no sibling manifest is legacy. The two documented v0.2
headers remain accepted for read-only analysis in a warned compatibility mode;
unknown columns or malformed values are rejected. Output stays flat,
unversioned, and relative by default. It invents no scan ID or manifest and
claims no durable provenance. Rescanning creates versioned output; legacy input
is never modified.

## Implementation testing expectations

Follow-up implementation must include:

- golden manifest/CSV fixtures, round-trip tests, and deterministic ordering;
- state-transition and failure-injection tests for complete, incomplete,
  failed, abandoned, and atomic-write behavior;
- strict negative validation of headers, rows, types, enums, IDs, paths, mixed
  scan IDs, hashes, and unsupported versions;
- both accepted v0.2 headers and privacy-safe legacy analysis;
- seeded absolute paths in errors, config, and symlink targets, asserting none
  reach versioned artifacts;
- file, directory, outside-root, broken, and cyclic symlinks with no target access;
- identity/fingerprint tests distinguishing IDs, stat changes, duplicates, and
  future checksums;
- analysis tests proving no access to source-library paths; and
- at least 100,000 generated rows to detect quadratic behavior and validate
  counts without committed library data or wall-clock assertions.

After review and merge, work may split into schema/validation, artifact
lifecycle, metadata/symlinks, analysis provenance, legacy compatibility, and
failure/scale testing. No such issues should exist before that review.

## Open questions and exclusions

There are no unresolved schema 1 questions. Review changes amend this contract.

This adds no MusicBrainz, network access, staging, retagging, renaming, moving, deleting, or destructive behavior.
